How the explorer works

A self-contained, single-file HTML explorer built on LineUp.js v4.16.0. No server, no framework, no API keys. All 755 rows from Final_Index.xlsx are baked into the HTML at build time. Two interaction modes give users both an overview and a deep-dive into any pillar.

Column modes

TypeColumnsWhen visible
Context language_name, location_name, Region, status Always — pinned left in all modes
Overview scores weighted_index + 5 pillar columns (*_Indicators_norm) Overview mode (default)
Sub-indicator scores Pillar-specific _contrib columns (one per sub-indicator) Pillar drill-down mode (click any pillar header)

Overview ↔ Pillar drill-down

Clicking any pillar column header (e.g. "Language Tech") swaps the 5 pillar columns for that pillar's sub-indicator contribution columns. Context columns and weighted_index stay. The "Overview mode" button in the header always returns to the full pillar view.

Live weight adjustment

A compact weight bar in the header contains one range slider per pillar. Dragging any slider proportionally rebalances the others so the total stays at 100%. The weighted_index column recomputes in-browser using: 

weighted_index = Σ (Pillar_Indicators_norm × weight_i / Σ weight_j)

No server round-trip — 755 rows recalculate instantly. A Reset button restores published default weights. The side panel score line always shows the current weight split.

Data pipeline — build_index.py

A standalone, fully reproducible Python script at index/build_index.py. Running it from CrisisVoices/index/ reads 03-outputs/Final_Match_Countries_Languages.xlsx and writes 03-outputs/Final_Index_v2.xlsx (100 columns, up from 77 in v1).

python build_index.py # or with custom paths: python build_index.py \ --input 03-outputs/Final_Match_Countries_Languages.xlsx \ --output 03-outputs/Final_Index_v2.xlsx

The script runs these steps in order:

  1. Derive drivers_count from the ACAPS string-list field.
  2. Apply per-column normalisation (direct min-max, inverted min-max, divide-by-100, direct inversion).
  3. Sum normalised sub-indicators → Pillar_Indicators; min-max → Pillar_Indicators_norm.
  4. Compute 20 *_contrib columns and verify the invariant before writing.
  5. Compute simple_index (equal-weight average) and weighted_index (default weights).
  6. Write 9-sheet Excel workbook + CSV.
Reproducibility guarantee: running build_index.py on an unchanged Final_Match_Countries_Languages.xlsx produces bit-identical output to Final_Index_v2.xlsx. All methodology decisions are encoded in the script, making future updates — new data vintages, added sub-indicators, revised weights — a single-file change.

Why contribution columns? The two-step normalisation

Why sub-indicator bars don't directly sum to the pillar score (and what we do about it)

A natural expectation is: if you add up the sub-indicator values in the drill-down, they should equal the pillar score shown in the overview. This is not possible with the raw normalised sub-indicator columns (*_inv_norm) because of the two-step normalisation used in the index pipeline:

  1. Each sub-indicator is individually min-max normalised to [0, 1].
  2. The normalised values are summed into an intermediate column Pillar_Indicators (range [0, N] where N = number of sub-indicators).
  3. That sum is min-maxed again across all 755 rows to produce Pillar_Indicators_norm (the pillar score in the overview, range [0, 1]).

Step 3 changes the scale, so the individual sub-indicator norms cannot directly sum to the pillar norm: Σ sub_inv_norm_i = Pillar_Indicators ≠ Pillar_Indicators_norm.

Solution — contribution columns (_contrib): For each sub-indicator and each row, a proportional contribution weight is computed:

contrib_i = (sub_inv_norm_i / Pillar_Indicators) × Pillar_Indicators_norm

Σ contrib_i = Pillar_Indicators_norm  ✓

As of v0.3, these columns (suffix _contrib, one per sub-indicator, 20 total) are precomputed by build_index.py and stored directly in Final_Index_v2.xlsx. The browser reads them from the CSV — no in-browser recomputation required. The side panel shows all three values per sub-indicator: raw → norm → contrib.

Limitation: The contribution values scale with the pillar score. A sub-indicator with sub_inv_norm = 0.9 that belongs to a low-scoring language (pillar = 0.3) will show a smaller bar than the same sub-indicator for a high-scoring language (pillar = 0.9). This is intentional — the column shows how much this sub-indicator contributes to the row's pillar score, not its absolute normalised value.

Design principle: At any point where a user is looking at a score — whether in the table or in the side panel — the formula used to produce it and the source of the underlying data should be one click or one scroll away. The side panel is the primary vehicle for this; the PROCEDURE.html document is the reference for analysts who want the full methodology.

Reliability scoring (reliability.py)

Each language–country row receives a reliability score (0–100%) computed as the weighted fraction of key raw input columns that are non-null. Weights follow pillar importance in the index.

Pillar (weight)Key raw columns checked
Crisis Risk (25%)INFORM Severity Index, Active_Risks_Count, drivers
Language Tech (25%)Recorded Hours, Validated Hours, Articles, Datasets, Models
Linguistic (20%)total_speakers, Linguistic_Diversity_Index_2017
Comms & Tech (15%)MCI_Index, ICT_Development_Index, Digital_Language_Support
Social & Econ (15%)GDP_2015, Literacy_Rate, IHDI

Formula

def compute_reliability(row):
    pillar_scores = {}
    for pillar_name, (cols, weight) in PILLAR_COLUMNS.items():
        present = sum(1 for c in cols if pd.notna(row.get(c)))
        pillar_scores[pillar_name] = (present / len(cols)) * weight

    reliability_pct = sum(pillar_scores.values())  # weighted 0–1

    label = (
        "High"     if reliability_pct >= 0.90 else
        "Moderate" if reliability_pct >= 0.70 else
        "Low"
    )
    missing = [c for _, (cols, _) in PILLAR_COLUMNS.items()
               for c in cols if pd.isna(row.get(c))]

    return {
        "reliability_pct":   reliability_pct,
        "reliability_label": label,
        "missing_fields":    missing,
        "affected_pillars":  [p for p, s in pillar_scores.items() if s < PILLAR_COLUMNS[p][1]]
    }

Reliability labels

High
≥ 90% of weighted key fields present
Moderate
70–89% of weighted key fields present
Low
< 70% of weighted key fields present
Zero-fill bias: The index pipeline uses .fillna(0) before aggregation. For risk metrics (INFORM Severity, Active_Risks_Count), this underestimates vulnerability — missing data is treated as "no crisis." The reliability explainer (Feature 3) must surface this directional bias explicitly.

Interaction logging (HCI research)

Every user action is appended to logs/session_{timestamp}.jsonl. One JSON object per line, written by a log_action() helper in streamlit_app.py. Disabled via DISABLE_LOGGING=1 env var.

Research purpose: These logs capture how decision-makers explore AI-mediated humanitarian data — which features they use, in what order, and what counterfactuals they probe. The logged sequences form the corpus for a future HCI analysis of AI-assisted sensemaking.

Visualization Research

Visualization design proposals

Six research-backed design ideas for presenting the LT4CR ranking. Each proposal is evaluated against three criteria: the primary goal of the view, the user type it serves best, and how the published evidence supports its effectiveness at this scale (755 pairs, 5 weighted pillars, inverted metrics, 4-region hierarchy).

Scale constraint: 673 languages cannot fit on a standard linear list. Any technique built for "a few dozen ranked items" requires adaptation. The proposals below are selected because they all have explicit strategies for handling hundreds or thousands of items — via aggregation, clustering, or progressive disclosure.

Design A
Stacked Multi-Attribute Ranking
LineUp-style
Goal Make visible why a language–country pair ranks where it does — each pillar's contribution to the final score is a separate colored segment in the same row bar. Users can see at a glance whether a high score is driven by crisis risk, language tech absence, or a combination.
Users NGO analysts comparing multiple pairs; methodology reviewers auditing pillar weights; decision-makers testing "what if I weight crisis higher?" scenarios.
Supporting papers LineUp: Visual Analysis of Multi-Attribute Rankings — Gratzl, Lex et al., IEEE VIS 2013. Introduces stacked bar columns, sorting, and live weight adjustment as the canonical technique for composite rankings. Taggle: Combining Overview and Details in Tabular Data Visualizations — Stitz & Lex, 2016. Direct LineUp successor; adds hierarchical aggregation for handling thousands of rows (key for 755 pairs). Ranked-List Visualization: A Graphical Perception Study — ACM CHI. Empirically tested scrolled bars, treemaps, and wrapped bars; found wrapped bars optimal for single-item evaluation and global distribution reading.
Example lineup.js.org — interactive multi-attribute demo. Also see the JKU VDS Lab LineUp overview.
LT4CR adoption Each of the 755 rows shows a master bar (weighted_index) split into 5 colored segments proportional to each pillar's normalized score × weight. Inverted metrics (Comms & Tech, Language Tech) must be visually normalized — a low Common Voice count produces a tall bar segment, indicating vulnerability. Sidebar weight sliders recalculate all bars in real time. Fisheye zoom or pagination handles the 755-row list without overplotting.
Design B
Hierarchical Collapsible Table
Taggle-style
Goal Let users navigate the 4-region → 48-country → 755-pair hierarchy progressively. Collapsed rows show an aggregate risk glyph; expanding reveals the next level. Users never see all 755 rows at once unless they choose to.
Users Regional program officers scanning which countries need attention first; funders who think geographically rather than by language family.
Supporting papers Taggle: Combining Overview and Details in Tabular Data Visualizations — Stitz & Lex, 2016. Designed explicitly for large, complex tables; hierarchical aggregation with per-level visual representations is the key contribution. Hierarchical Aggregation for Information Visualization — Elmqvist & Fekete, IEEE TVCG 2010. Names the interaction vocabulary: drill-down / roll-up, local aggregation control, visual budget — deciding how many items render at once. Combining Overview and Details in Tabular Data Visualizations — cited in NSF survey. Shows nesting granular details inside semantic hierarchies preserves data relationships better than flat sorting.
Example caleydo.org/tools/taggle — live demo of the Taggle system. Also see INFORM Risk Index platform for a real-world nested country-risk layout.
LT4CR adoption Collapsed state: 4 region rows, each showing a distribution bar of weighted_index across its language pairs. Expanding a region: 48 country sub-rows with aggregate median score. Expanding a country: individual language–country pairs with the stacked pillar bars from Design A. This approach means users who only care about "Africa" never render the 400+ Asian language pairs. In Streamlit, implement with st.expander per region/country, lazy-rendering the inner dataframe.
Design C
Sortable Heatmap Matrix
Language × Country
Goal Show the many-to-many structure of the dataset: which languages span multiple countries, and whether the same language has very different risk scores depending on country context. Enables pattern detection across both dimensions simultaneously.
Users Researchers studying cross-border language dynamics; analysts looking for country-level clusters (e.g., "all East African languages score similarly").
Supporting papers Ranked-List Visualization: A Graphical Perception Study — ACM CHI. Evaluated treemaps as surprisingly accurate for reading individual cell values — relevant since a heatmap is a structured treemap variant. Hierarchical Aggregation for Information Visualization — Elmqvist & Fekete. Matrix views sorted by hierarchical clustering are the recommended fallback when bipartite connections don't map to a clean tree.
Example Observable — D3 Clustered Heatmap — sortable rows and columns via hierarchical clustering. Also see Vega-Lite heatmap example.
LT4CR adoption Rows = languages (up to 673); columns = countries (48); cell color = weighted_index on a green→red scale; blank cell = this language not present in this country. Sort rows by average score across countries, or by language family. Sort columns by regional grouping (4 region bands as column separators). Clicking a cell opens the Tab 3 language detail panel. At 673 × 48, most cells will be blank (only 755 / (673 × 48) ≈ 2.3% fill rate), so the matrix is sparse — that sparsity is itself informative (most languages are country-specific, not cross-border).
Design D
Bipartite Sankey Flow
Language ↔ Country
Goal Expose the geographic spread of cross-border languages — Arabic, French, Spanish, Swahili each spoken in multiple countries with varying risk scores. The view answers: "where does this language appear globally and how does its risk vary by country context?"
Users Language policy researchers; cross-border humanitarian coordinators planning multilingual crisis communication campaigns.
Supporting papers The Science of Visual Data Communication: What Works — Franconeri et al., Psychological Science in the Public Interest, 2021. Documents how continuous paths and flow links reduce cognitive friction when tracking multi-tiered dependencies — the core perceptual justification for Sankey. FairSpace: An Interactive Visualization System for Multi-Attribute Rankings — Wiley. Reviews Podium and FairFuse; discusses bipartite layouts as a way to decompose where ranking inequities originate.
Example SankeyMATIC flow builder — for prototyping flow layouts. For a production implementation, see Plotly Sankey diagrams (already in our stack).
LT4CR adoption Left column: 48 country nodes, color-coded by region. Right column: 673 language nodes, collapsed into "Others" unless a language appears in ≥ 3 countries or is selected via search. Ribbon thickness = weighted_index of the pair. Clicking a language node dims everything else and highlights all country connections, revealing its global spread (Dual-End Highlighting). Plotly's built-in Sankey supports this with go.Sankey — already in our dependency tree. The main challenge is performance: 755 ribbons at once may be slow; a Top-20-by-score default filter is needed with a "show all" toggle.
Design E
Regional Equity View
FairSight-style
Goal Ask whether the ranking distributes attention equitably across the 4 regions. Is Africa systematically over- or under-represented in the top-50 highest-risk pairs relative to its share of the dataset? This reframes the question from "here is the ranking" to "here is whether the ranking is fair across regional lines."
Users Funders checking for structural bias in how the index allocates priority; researchers studying whether the composite metric inadvertently penalizes regions with better reporting (more data → lower apparent risk).
Supporting papers FairSight: Visual Analytics for Fairness in Decision Making — cited in FairSpace survey. Introduces Group-Fairness DCG: whether top-k positions distribute proportionately across predefined groups (regions, in our case). FairSpace: An Interactive Visualization System for Multi-Attribute Rankings — Wiley, 2022. Reviews state-of-the-art interactive systems (Podium, FairFuse) that enforce fairness on multi-attribute datasets. Interactive Explainable Ranking — ACM CHI, April 2026. Formalizes explainable ranking tasks reflecting real-world use cases; includes fairness as a first-class concern alongside explanation.
Example INFORM Risk Index platform — regional distribution panels. For the distribution chart style, see Plotly violin / ridgeline plots.
LT4CR adoption A side-by-side panel showing the distribution of weighted_index scores per region (violin or box plots). A Group-Fairness metric answers: of the top-k highest-risk pairs, what fraction comes from each region, and does that match each region's proportional share of the 755 pairs? A possible finding: if most top-50 pairs are in Asia but Africa has 40% of the dataset, the index may be under-flagging African language risk — possibly because ACAPS data is sparser for Africa (stable-country null effect). This connects directly to the reliability scoring system already planned.
Design F
Inferred Weight Discovery
Podium-style
Goal Let stakeholders reveal their implicit priorities by example — drag ~10 language pairs into an order that matches their intuition, and the system backs out what pillar weighting is consistent with that ordering. This turns weight-setting from a guessing exercise into a reflective, evidence-grounded conversation.
Users NGO directors or funders who have strong domain opinions about which communities matter most but cannot translate those opinions into a percentage breakdown. Also valuable as an HCI research probe: what implicit values do users reveal when they order pairs by intuition?
Supporting papers Podium: Ranking Data Using Mixed-Initiative Visual Analytics — Wall, Das et al., IEEE TVCG 2018. Rather than asking users to specify weights directly, Podium infers a Ranking SVM model from user-provided example orderings, then explains which attributes the implied weighting depends on. Interactive Explainable Ranking — ACM CHI, April 2026. Evaluates systems on real-world ranking tasks; the mixed-initiative paradigm (user provides examples, system infers model) is identified as the most effective interaction mode for non-expert users. LineUp: Visual Analysis of Multi-Attribute Rankings — Gratzl et al., IEEE VIS 2013. The complementary direction: while LineUp starts from weights and shows the ranking, Podium starts from a ranking and recovers the weights.
Example Podium on GitHub — the original research prototype. For the Ranking SVM logic, see the SVM-Rank library.
LT4CR adoption Present the user with ~10 randomly-sampled language pairs. They drag them into a priority order. A backend call runs a simple Ranking SVM (scikit-learn LinearSVC on pairwise preference vectors) across the 5 pillar scores to recover the implied weights. The result is displayed as: "Your ordering implies Crisis Risk matters 3× more than Language Tech, and Social & Economic is treated as nearly irrelevant." The user can then accept this weighting (it populates the sidebar sliders) or adjust. This doubles as an HCI research instrument: log the user's intuitive ordering alongside their inferred weights for post-session analysis.
Version History

v0.5.2 — Weight tool improvements shipped

Two improvements to the pillar weight controls, making the current weighting more visible and precise weight entry easier.

What changed from v0.5.1

AreaBeforev0.5.2
Weight composition strip No visual summary of the current weight distribution A full-width colored bar at the top of the weight section shows all five pillar weights as proportional segments. Updates live as sliders are dragged. Fixed (locked) segments show a diagonal hatching pattern.
Number inputs A read-only % label showed the current value — no way to type an exact weight Each pillar now has an editable number input alongside the slider. Type a value and press Enter (or Tab) to apply it; redistribution among unfixed pillars happens immediately.
Fix/lock weights Moving any slider redistributed weight across all other pillars A fix button per pillar locks its weight. Locked pillars are skipped during redistribution; the moved slider is clamped so the sum always stays at 100%. Reset clears all locks and restores defaults.

v0.5.1 — Data reliability panel superseded

Adds two reliability-focused features on top of v0.5. No data changes, no structural changes — all computed from the same baked CSV at page load.

What changed from v0.5

AreaBeforev0.5.1
Pillar info card — Coverage column Card showed sub-indicator name, source, frequency, and normalization formula only A fifth column shows a fill bar + percentage of the 755 rows with non-null data for that sub-indicator. For inverted indicators (null → zero-fill → inverted), a ↑ over / ↓ under bias label (with tooltip) signals whether missing data overestimates or underestimates vulnerability.
Data reliability panel No dataset-wide reliability view existed A persistent Data reliability button (amber, top bar) opens a full-width panel below the header. The panel shows:
  • Summary row: High ≥90% / Moderate 70–89% / Low <70% row counts (weighted by pillar weight × fraction of sub-indicators present per row)
  • Per-indicator table grouped by pillar: overall coverage % with country count for country-level indicators (e.g. 37/48 countries), four mini fill bars for Africa / Asia / N. America / S. America, and bias direction
The regional breakdown immediately surfaces geographic gaps — e.g. Active Risks (ACAPS) is 0% for S. America and 19% for N. America because ACAPS only tracks active crisis contexts.
Reliability computation: Each row's reliability score is the weighted sum of per-pillar completeness fractions using the default pillar weights (Linguistic 20%, Crisis Risk 25%, Comms & Tech 15%, Language Tech 25%, Social & Econ 15%). A pillar contributes its full weight only if all its sub-indicators are non-null. The score is computed from default weights regardless of any user slider adjustments, since it reflects a structural property of the dataset.
